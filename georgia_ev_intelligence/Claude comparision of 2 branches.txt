     ---
     D. Problems Found in sreeja-arch

     D1. Retrieval Failure — Missing Company from Multi-Company Answer

     Root cause (confirmed from code):
     1. retrieve_and_rerank() → top_k=70, rerank_top_n=25 (default)
     2. If there are >25 matching companies, those after rank 25 are silently
     dropped
     3. Dense embedding does NOT guarantee that all semantically-similar chunks
     rank in top 70 — exact company-name matches can rank lower than narrative
     chunks
     4. No metadata filtering means the 70 candidates include off-topic chunks
     (wrong tier, wrong county)
     5. The cross-encoder at retrieval.py:113 reranks correctly but its input
     pool is already truncated

     Example: Question requires 3 companies → chunk for company #3 scores rank
     #26 after reranking → dropped → answer has only 2 companies.

     D2. Irrelevant Answer — Extra Fields Included

     Root cause:
     1. format_context() at retrieval.py:147 renders all fields: company_name,
     source_url, and full text blob (which contains tier, role, OEM, employment,
      etc.)
     2. Prompts in prompts.py have no instruction to restrict output to the
     asked field
     3. LLM reads the full context and dumps all fields it sees

     D3. No Query-Type Detection

     - sreeja-arch treats all questions identically (always dense retrieve →
     rerank → generate)
     - Counting questions ("how many"), list-all questions ("list all
     companies"), and aggregate questions ("which county has most employment")
     need different retrieval paths

     D4. Evaluation Weight Inconsistency

     - sreeja-arch ragas_runner.py: faithfulness=0.25, answer_correctness=0.15
     - new-Arch evaluator.py: faithfulness=0.20, answer_correctness=0.30
     - These produce incompatible final_score values — can't compare across
     branches directly

     ---
     G. Final Recommended Architecture for sreeja-arch

     Given that sreeja-arch's purpose is an LLM comparison experiment (not a
     production agent), the recommended architecture is:

     1. Retrieval: Dense Qdrant + cross-encoder rerank (keep), but:
       - Increase top_k to 120
       - Add a lightweight query classifier to detect list/count questions
       - For list questions: increase rerank_top_n to 50 or scan all results
     above a threshold
       - Enable hybrid (dense+BM25) using vector_store.py's existing hybrid
     search capability
     2. Context formatting: Restructure to structured field-per-line format (fix
      E3 above)
     3. Prompts: Add field-focus instruction + exhaustive-list instruction (fix
     E2)
     4. Evaluation: Keep official ragas + Kimi 2.6 (more reproducible than local
      judge)
     5. For production (not comparison): Adopt new-Arch's deterministic entity
     extraction + SQL path for structured queries; keep dense vector only as
     fallback for narrative questions

     ---
     H. Remaining Risks

     1. Top-N still truncates: Even at rerank_top_n=50, if there are 60
     companies matching a condition, 10 will be dropped. True fix requires
     scan-all for list queries.
     2. Kimi 2.6 API dependency: Evaluation fails if Kimi API is down or key
     expires. No fallback.
     3. nomic-embed-text mismatch: If the indexed chunks used a different
     tokenizer or model version, query embeddings may not align perfectly.
     4. LLM non-determinism: temperature=0 is set but Ollama models may still
     produce slight variation across runs.
     5. Tavily for rag_pretrained_web: External API — rate-limited and costs
     credits.

     ---
     I. Next Experiments to Run

     1. Enable hybrid search: Turn on dense+BM25 in retrieval.py using
     vector_store.py's existing hybrid capability. Measure recall improvement on
      list questions.
     2. Add SQL path for sreeja-arch: For questions matching "list all", "how
     many", "which county" patterns, bypass vector search and run a pandas query
      directly on the GNEM Excel. Compare recall.
     3. Compare judge quality: Run the same 50 questions through both new-Arch
     (Ollama qwen2.5:14b judge) and sreeja-arch (Kimi 2.6 judge) and compare
     scores. Are they correlated?
     4. Answer correctness breakdown by category: Check if certain question
     categories (AGGREGATE, RISK, CYPHER-TIER) consistently underperform. Focus
     retrieval fixes there.
     5. Test larger context window: Try num_ctx=16384 for gemma3:27b to see if
     longer context helps list questions.
